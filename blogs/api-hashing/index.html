API hashing

0xCyberGenji | Jun 9, 2024 min read

Merhabalar. Bu yazıda, maldev alanındaki API Hashing yöntemini ele alacağım.

API Hashing, İçe aktarılmış (Imported) Windows API’ların taşınabilir ve yürütülebilir dosyalarda bulunmasını zorlaştırmak için kullanılan bir yöntemdir. Bu yöntem, Windows API’larının adlarını ve adreslerini gizlemek için kullanılır. Bu yazıda ise temelden buna odaklanılacaktır.

PE (Portable Executable) Yapısı Nedir?

Her şeyden önce PE (Portable Executable) yapısını anlamanın gerekli olduğunu düşünüyorum çünkü bu tekniğimiz PE içerisindeki bir tablo üzerine.

PE, Windows işletim sistemlerinde 32 ve 64 bit sürümlerde ve UEFI ortamlarda kullanılan executable dosyalar, object kodlar, DLL’ler ve diğerleri için bir dosya biçimidir. PE biçimi, Windows İşletim sistemi loader’in yürütülmek istenilen kodu yönetmesi için gereken veri yapıları içermektedir.

Bir PE dosyası, dinamik linker’a (dynamic linker) yürütülebilir dosyanın dosya belleğin nasıl eşleceğini ve çalıştırılacağını belirten bir dizi başlık ve bölümden oluşur. Bu başlıklar ve bölümler, işletim sisteminin dosyayı doğru şekilde yüklemesine ve çalıştırmasına olanak tanır.

Portable Executabe'ın Yapısı

PE yapısında bulunan bazı önemli başlıklar şunlardır:

  • DOS Header (IMAGE_DOS_HEADER): Bu başlık, eski MS-DOS işletim sistemleri tarafından yürütülebilir dosyanın tanınması için gerekli minimum bilgileri içerir. Aşağıdaki bilgileri içerir:

  • DOS Stub: Bu, DOS işletim sistemi tarafından yürütülebilir dosyanın tanınması için gereken minimum bilgileri içerir. Bu, dosyanın yürütülebilir olup olmadığını belirlemek için kullanılır.

  • PE Header (IMAGE_NT_HEADERS): Bu başlık, PE formatına özgü önemli bilgileri içerir. İşletim sisteminin dosyayı doğru şekilde yüklemesi ve çalıştırması için gerekli olan dosya boyutu, giriş noktası adresi, isteğe bağlı DLL’ler ve diğer bilgiler buradadır.

  • File Header (IMAGE_FILE_HEADER): Bu başlık, dosyanın genel bilgilerini içerir. Dosyanın mimarisi, dosya türü, bölümler ve diğer bilgiler burada bulunur.

  • Optional Header (IMAGE_OPTIONAL_HEADER): Bu başlık, dosyanın yüklenmesi ve çalıştırılması için gerekli olan bilgileri içerir. Bu başlık, dosyanın yüklenmesi ve çalıştırılması için gerekli olan bilgileri içerir. Bu başlık, dosyanın mimarisi, dosya türü, bölümler ve diğer bilgiler burada bulunur.

EAT (Export Address Table) ve IAT (Import Address Table) Nedir?

EAT ve IAT, PE dosyalarında bulunan iki önemli tablodur. Bu tablolar, dosyanın yürütülmesi sırasında işletim sistemi tarafından kullanılan fonksiyonların adreslerini ve adlarını içerir.

  • EAT (Export Address Table): Dışa Aktarım Adres Tablosu olan EAT, bir PE dosyasının dışarı aktardığı fonksiyonların adreslerini ve adlarını içeren bir tablodur. Bu tablo, dinamik linker’ın (dynamic linker) yürütülebilir dosyayı yüklerken ve çalıştırırken hangi fonksiyonların kullanılabileceğini belirlemesine yardımcı olur. Bunu özellikle aklınızda tutun çünkü ileride bu tablo üzerinde işlem yapacağız.

  • IAT (Import Address Table): İçe Aktarım Adres Tablosu olan IAT, bir PE dosyasının diğer PE dosyalarından içe aktardığı fonksiyonların adreslerini ve adlarını içeren bir tablodur. Aynı şekilde bu tabloda dinamik linker’ın yürütülebilir dosyayı yüklerken ve çalıştırırken hangi fonksiyonların içe aktarılması gerektiğini belirlemesine yardımcı olur.

API Hashing Nedir?

Teorik olarak API Hashing, bir API fonksiyonunun adını veya diğer tanımlayıcı özelliklerini bir hash değerine dönüştüren bir tekniktir. Bu hash değeri, fonksiyonun kimliğini temsil etmek için kullanılır. Bu yöntem, API’ların adlarını ve adreslerini gizlemek için kullanılır.

Örneğin sadece MessageBoxA içeren bir projeyi derleyip çalıştırdığımızı varsayalım. Projemizde MessageBoxA kullandığımız için derleme sırasında compiler IAT tablosuna MessageBoxA API’in adresini ekleyecektir. Bu ise API’in bulunduğu modül yani DLL içerecektir.

Örnek üzerinden devam edersek şuan projede yer alan bir API’in IAT’a eklendiğini biliyoruz. Peki bunu nasıl gizleriz? API hashing tam da bu konu üzerine. Malware’imizi analiz eden kişi, malware’da kullanılan API’ları IAT içerisinden rahat bir şekilde görebilir. Bu bağlamda, projemizde bu API’lara farklı isimler vererek IAT’a eklersek, analiz eden kişi bu API’ları kolayca göremeyecektir. Temelde API hashing yöntemi budur. İlgili API’in adını değiştirip hashleyerek IAT tablosuna eklemek.

Yöntem tam olarak böyle. API Hashing, belirli bir modülün EAT’sinde dolaşarak daha önce hesaplanılan hash ile eşlesen bir API’yi bulmaktır. Adresini aldıktan sonra Fonksiyon Pointer aracılığıyla API’yi çağırabiliriz.

#include <stdio.h>
#include <Windows.h>

DWORD CalculateHash(char* data) {
    DWORD hash = 0x99;
    for (size_t i = 0; i < sizeof(data); i++) {
        hash += (hash * 0xab10f29f + data[i]) & 0xffffff;
    }
    return hash;
}

int main() {
    DWORD hash_value = CalculateHash("CreateThread");

    printf("CreateThread: 0x%00x\n", hash_value);
    getchar();
    return 0;
}

Yukarıdaki örnekte, verilen API’in hash değerini hesaplayan basit bir kod parçası hazırlanmıştır. İlk olarak API’nin adı alınır ve hesap yapacak CalculateHash değişkenine gönderilir.

Daha sonra bu değişken içerisinde hash adında ve 0x99 değer tutan bir DWORD değişkeni oluşturduk Bu değer, karma hesaplamasının başlangıç ​​noktası olarak kullanılır. Daha sonra bir döngü başlattık. Bu döngüde, mevcut hash değerini 0xab10f29f sabit değeri ile çarpılır. Bu, her karakterin karma değerine katkısını artırmaya yardımcı olur. Mevcut karakterin değeri, çarpılan değere eklenir. Sonuç, 0xffffff değeri ile bitsel AND işlemine tabi tutulur. Bu, karma değerinin 32 bitlik bir tamsayı olarak kalmasını sağlar. İlerideki süreçerde bu fonksiyonu kullanacağız.

API’in Adresini Bulma

Şimdi ise API’nin adresini bulma işlemine geçeceğiz. Adım adım windbg ile, modüllerin ve API’ların nasıl sıralandığını ve PE yapısını windbg ile göreceğiz ve ardından öğrendiklerimizi C projesine dökeceğiz.

İlk olarak Windbg aracılığıyla çalışan bir process’in PEB (Process Enviroment Block) yapısını inceleyerek, process’in modüllerinin taban adreslerine göz atacağız. Devam etmeden şunu da belirteyim PEB, bir process’in çalışma zamanı bilgilerini içeren bir yapıdır. Bu yapı, process’in modüllerinin taban adreslerini, heap ve stack bilgilerini, environment değişkenlerini ve diğer bilgileri içerir.

64 bit process’lerde PEB, Thread Enviroment Block (TEB) bloğundan 0x60 ofsette bulunur. TEB ise yürütülen thread’in çalışma zamanı bilgilerini içeren bir yapıdır. TEB yapısı GS segment kaydı tarafından işaretlenir. Bu bağlamda PEB yapısı da GS segment kaydı ile GS:[0x60] adresinden erişilebilir.

  • InLoadOrderModuleList: Yürütülebilir dosya yüklendiğinde modüllerin yüklenme sırasını temsil eder.

  • InMemoryOrderModuleList: Modüllerin belleğe yüklenme sırasını temsil eder.

  • InInitializationOrderModuleList: Modüllerin başlatılma sırasını temsil eder.

C projesine başlamadan önce aşağıdaki yapıları C projesine eklemeyi unutmayın:

typedef struct _UNICODE_STRING {
    USHORT Length;
    USHORT MaximumLength;
    PWSTR  Buffer;
} UNICODE_STRING, * PUNICODE_STRING;

typedef struct _LDR_MODULE {
    LIST_ENTRY InLoadOrderModuleList;
    LIST_ENTRY InMemoryOrderModuleList;
    LIST_ENTRY InInitializationOrderModuleList;
    PVOID BaseAddress;
    PVOID EntryPoint;
    ULONG SizeOfImage;
    UNICODE_STRING FullDllName;
    UNICODE_STRING BaseDllName;
    ULONG Flags;
    USHORT LoadCount;
    USHORT TlsIndex;
    LIST_ENTRY HashTableEntry;
    ULONG TimeDateStamp;
} LDR_MODULE, * PLDR_MODULE;

typedef struct _LDR_DATA_TABLE_ENTRY
{
    LIST_ENTRY InLoadOrderLinks;
    LIST_ENTRY InMemoryOrderModuleList;
    LIST_ENTRY InInitializationOrderModuleList;
    PVOID DllBase;
    PVOID EntryPoint;
    ULONG SizeOfImage;
    UNICODE_STRING FullDllName;
    UNICODE_STRING BaseDllName;
    ULONG Flags;
    USHORT LoadCount;
    USHORT TlsIndex;
    union
    {
        LIST_ENTRY HashLinks;
        struct
        {
            PVOID SectionPointer;
            ULONG CheckSum;
        };
    };
    union
    {
        ULONG TimeDateStamp;
        PVOID LoadedImports;
    };

    PVOID PatchInformation;

IDE içerisinde bu yapılar tanımlı olmadığı için elle kendimiz tamamlamamız gerekmektedir. Şimdi ise kodlamaya geçebiliriz:

#include "utils.h"

DWORD CalcHash(char* TargetModule) {
	/* 
		Bu kısım parametre olarak verilen modülün hash'ini bulmak için kullanılacaktır. 
	*/
	DWORD Hash = 0x99;
	for (int x = 0; x < sizeof(TargetModule); x++) {
		Hash += TargetModule[x] + (Hash << 1);
	}
	return Hash;
}

DWORD CalcModuleHash(LDR_MODULE* ModuleLink) {
	char*  ModuleName[64];
	size_t counter = 0x0;

	while (ModuleLink->BaseDllName.Buffer[counter] && counter < sizeof(ModuleName) - 1) {
		ModuleName[counter] = (char)ModuleLink->BaseDllName.Buffer[counter];
		counter++;
	}
	ModuleName[counter++] = 0;
	return CalcHash((char*)CharLowerA(ModuleName));
}

HMODULE GetModuleBaseAddress(DWORD Hash) {
	HMODULE ModuleBaseAddress = NULL;
	INT_PTR PEB					= __readgsqword(0x60);	// PEB'in offseti
	INT_PTR LDR				    = 0x18;					// PEB içerisindeki LDR'in offseti
	INT_PTR FlinkOffset			= 0x10;					// InLoadOrderModuleList'in offsetş

	INT_PTR PEB_LDR_DATA = *(INT_PTR*)(PEB + LDR);				  // PEB içerisinden LDR'e ulaş
	INT_PTR FirstFlink = *(INT_PTR*)(PEB_LDR_DATA + FlinkOffset); // LDR içerisinden InLoadOrderModuleList yapısına ulaş
	LDR_MODULE* LDR_DATA_TABLE_ENTRY = (LDR_MODULE*)FirstFlink;

	do {
		/* 
			Buradaki döngü, process'e yüklenen tüm modül listesi üzerinde işlem yapar.
			Yüklenen her modülün base address'i ve hash değeri alınır ve kontrol yapılır. 
		*/
		LDR_DATA_TABLE_ENTRY = (LDR_MODULE*)LDR_DATA_TABLE_ENTRY->InLoadOrderModuleList.Flink;
		if (LDR_DATA_TABLE_ENTRY->BaseAddress != NULL) {
			if (CalcModuleHash(LDR_DATA_TABLE_ENTRY) == Hash) {
				break;
			}
		}
	} while (FirstFlink != (INT_PTR)LDR_DATA_TABLE_ENTRY);

	ModuleBaseAddress = (HMODULE)LDR_DATA_TABLE_ENTRY->BaseAddress;
	return ModuleBaseAddress;
}

int main() {
	DWORD	GetAPIHash		= CalcHash("CreateThread");
	HMODULE	GetBaseAddress = GetModuleBaseAddress(GetAPIHash);

	if (GetBaseAddress == NULL) {
		printf("İlgili API'in Modül adresi bulunamadı!\n");
		return -1;
	}
	printf("Base Address: 0x%p\n", GetBaseAddress);
	getchar();
	return 0;
}

Bu kod parçasında, öncelikle CalcHash fonksiyonu ile API’nin hash değeri hesaplanmaktadır. Daha sonra CalcModuleHash fonksiyonu ile modülün hash değeri hesaplanmaktadır. Bu fonksiyon, modülün adını alır ve CharLowerA fonksiyonu ile küçük harfe dönüştürür. Daha sonra CalcHash fonksiyonuna gönderilir ve hash değeri hesaplanır.

Daha sonra GetModuleBaseAddress fonksiyonu ile modülün base adresi bulunmaktadır. Bu fonksiyon içerisinde PEB yapısını okuyoruz ve LDR_DATA_TABLE_ENTRY yapısına ulaşıyoruz. Daha sonra yüklenen modüllerin listesini dolaşarak, CalcModuleHash BaseDllName.Buffer içerisindeki tüm karakterleri alır ve input olarak alınan hash ile karşılaştırma yapar. Eğer eşit ise modülün base adresi döndürür.

Base Address’i bulduktan sonra API’in temel adresini bulmamız gerekiyor. Bunun için ihtiyacımız olan şey, IMAGE_EXPORT_DIRECTORY yapısıdır. Bu yapı, Export Address Table’ın (EAT) adresini içerir. Bu yapıda içeren bazı önemli alanlar şunlardır:

  • NumberOfFunctions: Export edilen fonksiyonların sayısı
  • AddressOfFunctions: Export edilen fonksiyonların adreslerinin bulunduğu tablonun adresi
  • AddressOfNames: Export edilen fonksiyonların isimlerinin bulunduğu tablonun adresi

Dikkatimizi 0x70 offsetine yani IMAGE_DATA_DIRECTORY yapısına çekiyoruz. Bu yapı, Export Address Table’ın RVA’sı (Relative Virtual Address) gibi Import Address Table için önemli bilgileri tutar.

IMAGE_EXPORT_DIRECTORY yapısı ise şu şekildedir:

typedef struct _IMAGE_EXPORT_DIRECTORY {
    DWORD   Characteristics;
    DWORD   TimeDateStamp;
    WORD    MajorVersion;
    WORD    MinorVersion;
    DWORD   Name;
    DWORD   Base;
    DWORD   NumberOfFunctions;
    DWORD   NumberOfNames;
    DWORD   AddressOfFunctions;     // RVA from base of image
    DWORD   AddressOfNames;         // RVA from base of image
    DWORD   AddressOfNameOrdinals;  // RVA from base of image
} IMAGE_EXPORT_DIRECTORY, * PIMAGE_EXPORT_DIRECTORY;

Bu yapıda bizim için önemli olan alanlar şunlardır:

  • AddressOfFunctions: Export edilen fonksiyonların adreslerinin bulunduğu tablonun adresi

  • AddressOfNames: Export edilen fonksiyonların isimlerinin bulunduğu tablonun adresi

  • AddressOfNameOrdinals: Export edilen fonksiyonların isimlerinin bulunduğu tablonun adresi

Şimdi ise bu bilgilerin ışığında API’in adresini bulmak için C projemizi güncelleyelim:

LPVOID GetAPIAddress(HMODULE ModuleAddress, DWORD ApiHash) {
	// Alınan Modülün IMAGE_DOS_HEADER offset'ini al
	PIMAGE_DOS_HEADER		DOSHeader = (PIMAGE_DOS_HEADER)ModuleAddress; 

	// Alınan DOS_HEADER'ın e_lfanew değeri ile IMAGE_NT_HEADERS offsetini al
	PIMAGE_NT_HEADERS		NTHeader  = (PIMAGE_NT_HEADERS)((LPBYTE)ModuleAddress + DOSHeader->e_lfanew);
	
	// NTHeader içerisinden  IMAGE_EXPORT_DIRECTORY al
	PIMAGE_EXPORT_DIRECTORY ExportDir = (PIMAGE_EXPORT_DIRECTORY)(
		(LPBYTE)ModuleAddress + NTHeader->OptionalHeader.DataDirectory[IMAGE_DIRECTORY_ENTRY_EXPORT].VirtualAddress); 

	/* IMAGE_EXPORT_DIRECTORY yapısı içerisinden Fonksiyon adresleri içeren yapıyı al (AddressOfFunctions) */
	PDWORD FunctionAddrs	= (PDWORD)((LPBYTE)ModuleAddress + ExportDir->AddressOfFunctions);

	/* IMAGE_EXPORT_DIRECTORY yapısı içerisinden Fonksiyon adları içeren yapıyı al (AddressOfNames) */
	PDWORD FunctionNames	= (PDWORD)((LPBYTE)ModuleAddress + ExportDir->AddressOfNames);

	/* IMAGE_EXPORT_DIRECTORY yapısı içerisinden Fonksiyon sırası içeren yapıyı al (AddressOfOrdinals) */
	PDWORD FunctionOrdinals = (PDWORD)((LPBYTE)ModuleAddress + ExportDir->AddressOfNameOrdinals);

	for (DWORD x = 0; x < ExportDir->AddressOfFunctions; x++) {
		/* 
			Bu döngüde AddressOfNames yapısından tek tek tüm isimler sırasıyla alınacak ve 
			bu isimerin hash'i alınacak. Ardından hesaplanan hash ile aranan API'in hash ile
			karşılaştırılacak. Eğer eşit ile API'in adresi döndürülecek. 
		*/
		LPSTR CurrentFuncName = (LPSTR)((LPBYTE)ModuleAddress + FunctionNames[x]);
		DWORD HashOfFunction = CalcHash(CurrentFuncName);
		if (HashOfFunction == ApiHash) {
			printf("%s: Hash Found!\n", FunctionNames[x]);
			return (LPVOID)((LPBYTE)ModuleAddress + FunctionAddrs[FunctionOrdinals[x]]);
		}
	}

	return NULL;
}

Bu kod parçasında, GetAPIAddress fonksiyonu ile API’in adresini bulmaktayız. İlk olarak, modülün IMAGE_DOS_HEADER yapısını alıyoruz. Daha sonra IMAGE_NT_HEADERS yapısına ulaşıyoruz. Bu yapıda IMAGE_EXPORT_DIRECTORY yapısını alıyoruz.

Daha sonra IMAGE_EXPORT_DIRECTORY yapısı içerisinden AddressOfFunctions ve AddressOfNames alanlarından fonksiyon adreslerini ve isimlerini alıyoruz. Daha sonra bu isimlerin hash’ini alıyoruz ve aranan API’nin hash ile karşılaştırıyoruz. Eğer eşit ise API’nin adresini döndürüyoruz.
